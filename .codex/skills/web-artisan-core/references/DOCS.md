# Docs mode

The standard is not “clean documentation.” The standard is a reading and retrieval instrument whose restraint is visibly authored.

## Information architecture

Define:

- audience levels and entry points;
- conceptual guides vs task guides vs reference;
- global navigation and local table of contents;
- versioning, language, product area, and status labels;
- search behavior and empty/no-result recovery;
- stable headings, anchors, and previous/next paths;
- mobile navigation and deep-link behavior.

## Typography and measure

- Body measure usually remains within a comfortable reading range; widen only for tables/code/diagrams.
- Create distinct type treatments for prose, headings, navigation, metadata, code, keyboard input, and API tokens.
- Ensure headings communicate structure without relying only on size.
- Tune line height, paragraph rhythm, list spacing, and code alignment in both themes.

## Code and technical content

- copy buttons with clear feedback;
- line wrapping/scroll policy chosen per code type;
- language labels and filenames where helpful;
- line highlighting and diffs with non-color cues;
- accessible syntax contrast in both themes;
- tabs only when alternatives are genuinely equivalent;
- tables that remain usable on mobile;
- diagrams with text alternatives or explanations.

## Quiet signature

Docs can have one restrained signature: a remarkable search, a beautifully legible API signature, a contextual example runner, a precise diagram language, or an unusually good navigation model. Do not add a marketing spectacle to prove the site was designed.

## Required states

Search loading/no results/error; version unavailable; deprecated content; auth-required content; long headings; deep nesting; huge tables; code overflow; copied state; focus/keyboard navigation; mobile menu; offline or failed content where applicable.
