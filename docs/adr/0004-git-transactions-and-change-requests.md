# ADR 0004: Git transactions and change requests

Status: accepted

## Context

Questions, claims, transcripts, and integrations all mutate the Git-backed database. They need concurrency and local-only behavior without assuming every operator can create remote pull requests.

## Decision target

Repository mutations acquire a repository lock, validate the expected lifecycle revision, write temp files, atomically rename, regenerate projections, and then release. A checkpoint service creates small state commits when configured; otherwise it leaves an explicit pending checkpoint visible in Maintenance.

Documentation integration uses an isolated Git worktree and an attempt-specific `ultradyn-attempts/<change-request-id>` branch. This namespace can coexist with manager-owned legacy `ultradyn/<question-id>` refs without renaming or deleting user refs. A change request is backend-neutral: local branches and review metadata are complete implementations; GitHub PR publication is optional. Remote polling never performs a review under a different identity—it creates a locally claimable review task.

## Consequences

The target design allows one repository to be served by multiple processes without silent lost updates. Users without GitHub write/review roles can complete checks locally; remote publication adds coordination, not core correctness.

## Implementation status

Question mutations use a canonical-repository-keyed machine-local lock, expected revisions, atomic replacement, and question-index regeneration. Raw artifact append is restartable across the file-publication/manifest boundary and fails closed when recovered bytes differ. Audio chunks likewise adopt byte-identical unmanifested bytes after metadata failure only from a nonblocking, no-follow descriptor bound to a regular file, rejecting symlinks, FIFOs, sockets, directories, path swaps, and byte conflicts. Audio-session schema version 2 records stable file identities and a pending append identity. Version-1 records migrate on read: every retained raw chunk must first pass stable no-follow inode, size, and digest validation, while historical completed records with `rawRetained: false` remain readable after legitimate raw cleanup. Before advanced sequence metadata is published, append securely creates `append-intent.json` with a transition-derived operation ID, whole-intent SHA-256 integrity, and exact prior, pending, and committed records; the same operation ID is durably present in pending session metadata. Recovery reads the journal through a stable no-follow regular-file descriptor and may roll back only metadata carrying that exact pending identity. Once committed metadata clears the pending marker, a later or forged journal can only be discarded and can never reinterpret an acknowledged chunk.

The held descriptor remains bound through publication. Verification completes every descriptor read and stat before making a final sequential no-follow pathname observation. Append clears the pending marker, removes the journal, and repeats that descriptor-then-path observation as the last filesystem operation before acknowledgement; a detected swap restores prior metadata and returns no acknowledgement. Recording-session reads and byte-identical acknowledgement retries revalidate committed chunks rather than trusting `nextSequence` alone. Matched-ask attachment reserves all raw paths in an atomic machine-local journal before publication and records a base-revision-scoped operation marker in portable provenance; retry therefore completes question, chat, record, provenance, and index exactly once, while a changed same-revision payload fails closed. Explicit asker rejection uses the same locked/journaled pattern to bind one reserved rejection artifact to one canonical decision-and-reopen update. The journal persists the exact actor and reason; completed recovery verifies the artifact manifest and bytes, all three provenance-event authors and relationships, and an operation identity recomputed from the exact reason, actor, asker, question, and merged base revision before cleanup. The base revision makes an identical later-attempt rejection distinct while concurrent duplicates and crash recovery stay exactly once. `decideAsker` accepts only acceptance or timeout; every rejection must use that canonical operation. Claiming a reopened question resets its rejecting asker to pending for the new attempt while provenance retains the prior rejection. Recovery composes onto a newer valid matched-ask record revision without overwriting intervening changes and rejects an impossible rewind.

Local documentation proposals use isolated worktrees, actual diffs, mandatory evaluator checks, and explicit approvals. Each attempt persists a deterministic fingerprint of its question, verbatim chat, ordered goals, structured answer, and proposed files/content. The production service always submits that current tuple through `ensureCurrent`; an existing proposal's stored files remain canonical unless a caller explicitly supplies a new proposal, so an Integrator is not rerun merely to compare nondeterministic output. Identical input reuses the active attempt; changed input creates a unique request/branch/worktree and terminally supersedes the prior active record without deleting its dirty worktree. Creation and supersession share a machine-local lock and durable operation journal written before Git mutation. Before that journal is persisted, creation validates the prior record's metadata identity, evaluator-input fingerprint, merge intent, eligible state, terminal markers, and complete relationship to the new operation. Recovery repeats that validation, accepts an already-applied supersession only when every timestamp and `supersededBy` value exactly bind the journaled record, and preflights the target metadata file, branch, ref registration, worktree path, symlinks, and recorded resource ownership before proposal materialization. Exclusive worktree creation is recorded back into the journal before the first recoverable checkpoint; an unrecorded pre-existing target is always a collision, even when it points at the expected base. Invalid input, invalid prior state, and target collisions leave refs, worktrees, metadata, and operation files byte-for-byte unchanged. Restart then reconciles journal, worktree, new metadata, old supersession, and cleanup boundaries before reads expose records; dirty journaled worktrees fail closed without deleting bytes. Merged attempts remain historical and a reopened question can complete another full attempt.

Change-request schema version 2 records exact evaluator input. Version 1 records are migrated on read: active and merged history remains readable, but unavailable historic question/chat/goals/answer fields are represented explicitly as `legacy-unavailable`, never fabricated, and evaluation, approval, or merge fails closed until a fresh attempt is created. An approved merge persists the exact base/branch intent before Git advances HEAD. Git first calculates the canonical merge result tree from the immutable reviewed branch head and prepared base without mutating either ref. The intent's SHA-256 authorization binding covers that expected tree together with the record and question identities, evaluator-input fingerprint, reviewed diff/base/branch heads, exact checks and approvals, merge actor, and start time. Every metadata read rejects intent unless the record is in an authorized merge/recovery state, has exactly one passed result from every mandatory evaluator, has no failed check, has an answerer or maintainer approval, and still matches that binding. Reconciliation revalidates the authorization rather than treating attacker-authored intent as authority. On restart it inspects current base HEAD first: an already-created merge must have exactly the authorized first and second parents and exactly the expected result tree. Once that exact commit exists, metadata and owned-worktree cleanup complete independently of any later attempt-ref movement. Only the pre-mutation path rechecks the mutable attempt ref after final base-worktree inspection, then invokes Git with immutable `branchHeadSha`, never the branch name. A failed Git merge is aborted back to persisted `baseHeadSha`; a forged two-parent commit, an unrelated HEAD, or a different result tree never advances metadata or cleanup and is never reset as unrelated state. Cleanup inspects tracked and untracked status, refuses dirty worktrees without deleting bytes, and removes clean worktrees without `--force`; the durable merge intent makes explicit cleanup retry safe without a second merge. The checkpoint setting is enforced around approved local merges: when enabled it commits managed question/settings state, and when disabled any uncommitted managed paths produce one explicit pending-checkpoint task and count in Maintenance. A general checkpoint service for portable mutations outside that merge path is still absent. Change-request metadata is machine-local, general map regeneration and automatic rebase/re-plan are absent, and cross-process safety is not established for every settings/Git workflow. GitHub polling creates visible review/re-review tasks, but they are not claimable/actionable and publication is not wired to local approvals. These are in-repository items in `BLOCKED_TASKS.md`.
