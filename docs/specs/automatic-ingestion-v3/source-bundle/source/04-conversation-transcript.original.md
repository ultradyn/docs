## Claude

Good brief — the end-to-end shape is clear. Let me take the indexing question first, then push back on where I think the actual constraint is, because I don't think it's indexing.

**Indexing.** For a git-based markdown corpus queried by an LLM agent, there are four realistic approaches: committed vector embeddings, a committed lexical index, an ephemeral (gitignored, rebuilt-from-repo) lexical index like tantivy/BM25, and pure agentic navigation — the Claude Code pattern of grep/glob plus human-readable map files (question index, cross-link maps, per-directory summaries) that are themselves just markdown or JSONL in the repo.

Goals: **G1** git-friendly (line diffs, no binary churn — your compression concern), **G2** retrieval quality for the agent, **G3** low ops complexity, **G4** semantic matching (finding prior questions worded differently).

| Idea | All | G1 | G2 | G3 | G4 |
|---|---|---|---|---|---|
| Committed vector DB | ✘ | ✘ | ✔ | ✘ | ✔ |
| Committed lexical index | ✘ | ? | ✔ | ✘ | ✘ |
| Ephemeral BM25 (derived, gitignored) | ✘ | ✔ | ✔ | ✔ | ✘ |
| Agentic nav over readable maps | ✔ | ✔ | ✔ | ✔ | ✔ |

The key move is treating derived indexes as build artifacts, never as committed state. The *committed* "index" is just plain-text maps: `questions/index.jsonl` (JSONL is git's friend — append-mostly, one record per line, clean diffs), a topic map, per-directory `_map.md` summaries. The semantic matching problem (G4) dissolves because the agent reads the question index and does the paraphrase-matching itself — LLMs are the best semantic matcher you have access to, and it's already in the loop. If lexical search over a large corpus becomes slow, add ephemeral BM25 as a tool the agent can call, rebuilt locally from HEAD. Embeddings buy you nothing here except an ops burden and a binary blob that fights git.

This also means "the indexing problem" mostly reduces to "keep the maps accurate," which is just another consistency obligation on the answer-integration step — which brings me to the criticism.

**The ID scheme.** Two problems with hash-of-question, first 7 hex chars. First, 7 hex = 28 bits, so by birthday math you hit 50% collision odds around ~19k questions — not hypothetical for a long-lived project. Use 12+ chars, and since inserts go through the index anyway, check-and-extend on collision costs nothing. Second and more important: deriving the ID from the question text means editing the question (typo, clarification) changes its identity, breaking every reference. And determinism buys you nothing — near-duplicate questions are worded differently, so they won't hash-collide anyway; dedup has to be semantic regardless. So just use a random or time-sortable ID (ULID is nice: sortable, no coordination). Identity should be assigned, not derived.

**Where I think the real constraint is.** You named indexing as the sticking point, but retrieval over a well-structured markdown repo is a solved-enough problem. The step that can actually sink the system is: *"the agent integrates the answer with the documentation... anywhere the documentation touches that is relevant... so the entire base remains consistent and complete."* That's an LLM making multi-file edits across a corpus with a consistency guarantee attached. Silent doc corruption, confident wrong edits, and drift accumulate there, and errors compound because future answers are grounded in past integrations.

The good news is your git choice already contains the fix, if you make it load-bearing: the integration step's output should be a **branch/PR, never a direct commit to main**. The answerer's final act isn't dictating — it's reviewing a diff. That bounds the blast radius of any LLM error to one reviewable changeset, gives you rollback for free, and handles concurrent answerers through ordinary merge workflow instead of a bespoke locking scheme. The agent's criticism job and integration job both become "propose," with a human holding decisive accept/reject.

Two smaller things. Breadth-first needs a decisive stopping criterion or the criticism-LLM will rabbit-hole exactly as you fear — and there's a clean one available: an answer is *done* when it addresses the question as asked with no contradiction against existing docs, and any depth gap the critic finds is not a blocker but **spawns a child question with its own ID into the unanswered queue**. Then breadth-first behavior falls out of the queue structure mechanically instead of depending on the LLM's judgment about when to stop digging. That also makes the question tree you described emerge for free.

And keep raw audio out of git proper — transcripts as text in-repo, audio blobs in LFS or an object store keyed by question ID, or the repo bloats fast.

Net: the concept is sound and the git substrate is the right call — but its value comes from using git's review mechanics as the safety rail on LLM edits, not just as storage. Indexing you can do the boring way.

## Claude

The asker-simulation check is the strongest addition here, and it works precisely *because* you're keeping the verbatim question. "Good enough" is defined relative to the asker's context, and the cleaned/structured version of a question tends to launder that context away — it becomes the question the system understood, not the question the person had. So the check should run against the raw question plus the surrounding chat log, not the structured form. One caveat on trusting it: an LLM roleplaying the asker shares none of the asker's tacit knowledge, so it'll catch "this doesn't address what was literally asked" reliably but will miss "this addresses what was asked but not what I actually needed." That's fine as long as manual mode can ping the real asker for closure on questions that matter — the simulated check is the cheap gate, the real asker is the decisive one.

On the tags: I'd generalize "raw vs generated" into a provenance field rather than a boolean, because generated questions *do* have a grounding document — it's just a different kind. A raw question's provenance is the verbatim recording plus chat log; a generated question's provenance is the parent question ID plus the specific critic finding that spawned it. Recording that finding matters: whoever eventually answers a deferred question needs to know *why the critic thought it was a gap*, otherwise they're answering a question stripped of its motivation, which is the same context-loss problem again one level down.

For the "maybe-unnecessary / extra detail" tag — worth having, but notice it's a priority judgment made at spawn time, which is when you have the least information about whether it matters. The honest signal comes later: if a deferred question is real, someone eventually asks it. So make that the promotion mechanism — when a new raw question comes in, the agent checks it against the deferred queue, and a semantic match links the two and bumps the deferred question to active with the new asker attached. You get deduplication and demand-driven prioritization out of a single mechanism, and the spawn-time tag becomes what it should be: a weak prior, not a decision.

One caution on the PR-review agent and the change summary. The summary relayed to the answerer should be generated from the diff alone, by an agent with a fresh context that hasn't seen the integration agent's reasoning or plan. If the summarizer shares context with the writer, it inherits the writer's blind spots and will happily summarize what the writer *intended* to do rather than what the diff actually does — and divergence between those two is exactly the error class you're building the check to catch. Same principle for the review agent generally: independent context, sees only the question, the answer, and the diff.

Audio in a gitignored local folder is right — just name the files by question ID so they're recoverable if a transcript ever looks suspect and you need to re-check against the source.

## Claude

On asker rejection: treat it as more than feedback — a rejection reopens the question, and the rejection reason is raw material of the same kind as an original question. Record it verbatim in the raw folder, attach it to the question's provenance chain, and now the next answer attempt is grounded in "here's what was tried and here's the asker's own words on why it missed." That also resolves the caveat from before about the simulated asker: the simulation is the cheap pre-filter, the real asker's accept/reject is the decisive check, and a rejection automatically feeds the loop rather than dying in a comment box.

On priority, one design warning that I think you'll see coming given the framework: since priority is "a combination of many factors," the tempting implementation is a weighted score — depth × 0.5 + recency × 0.3 + whatever. Don't. The factors aren't commensurable and the arithmetic will produce confident nonsense that's hard to argue with precisely because it's a number. Better: a small set of ordered tiers assigned by *rules*, where each rule is individually criticizable. Contradiction-spawned → top tier, always — because a contradiction isn't just "more important," it's qualitatively different: it means the doc base contains at least one wrong claim and everything touching both sides is suspect until resolved. It's decisive, not weighty. Depth decay is the default rule below that. And your human-override point is exactly why tiers beat scores: "this is tier 2, should be tier 1" is a judgment a person can make and justify in one sentence; "this is 0.63, should be 0.71" is not a conversation anyone can have. One number to check is only a cheap error-check if the number has meaning a human can dispute.

The MCP-agents-in-disguise pattern is right, and it composes with everything else you've built if you take it one step further: **the agent definitions live in the repo**. Each specialized agent is a file — prompt, allowed tools, input/output contract — versioned alongside the docs it operates on. That gets you three things. Changing an evaluator goes through the same PR discipline as changing docs, so "we made the critic more lenient" is a reviewable, revertable diff rather than a silent config change. Fresh context per call comes for free since each MCP invocation instantiates from the file. And the contract part matters most: each agent should have a narrow, structured output schema — the critic returns findings, the summarizer returns a diff summary, nothing returns freeform advice to another agent — because structured outputs are what keep a pipeline of LLMs from turning into a game of telephone. The specialization you're after isn't just a focused prompt; it's a focused *interface*, where staying on task is enforced by what the agent is allowed to emit, not just what it's asked to do.

One consequence worth noticing: at this point the repo contains the docs, the questions, the provenance, *and* the machinery that maintains all of it. The whole system state is one clone. That's a genuinely nice property — worth protecting as a design invariant when features get added later.

## Claude

The goal field is the piece that makes everything else click, and notice what it actually is: you've just put IGC into the schema. Goals across the top, the answer as the idea, and the critic's evaluation stops being "is this good quality?" (vague, undecidable) and becomes a row of decisive checks — does this satisfy implementation? ✔. Security analysis? ✘, no threat model addressed. That ✘ spawns a child question that inherits *only the unsatisfied goal*, which is your goal-dropping mechanism falling out naturally: scoped questions carry scoped goal lists because they're born from specific ✘ cells. And "good enough" finally has a referent — good enough *for these declared goals*, not good enough in the abstract, which was the unresolvable version. The simulated-asker check sharpens too: "pretend you asked this with goal: API integration" is a much better-posed roleplay than reconstructing intent from a transcript.

One practical suggestion: keep the goal vocabulary in the repo as a small, growing controlled list (implementation, API-integration, security-review, complexity-analysis...) with freeform allowed but nudged toward existing tags. Not for bureaucracy — for the agents. "Security-review" meaning the same thing across five hundred questions is what lets a critic apply consistent per-goal guidelines, and it's what makes your P1–P5 guideline-fitting workable, since priority rules can reference goal types ("contradictions P1; unsatisfied implementation goals on active questions P2..."). P1–P5 with fitted guidelines is exactly right, no further argument from me.

On the self-modifying repo and the agent-that-makes-agents: I'm on board, with one asymmetry to respect. A bad doc edit is a visible, recoverable error — someone reads the wrong doc, complains, you revert. A bad *agent* edit is generative: a critic that got 10% more lenient corrupts every answer that flows through it afterward, silently, and you discover it weeks later in the docs it waved through. Errors in the machinery compound; errors in the content don't. So the PR discipline should be tiered: auto-mode is fine for doc changes, but agent-definition changes default to manual review, and — this is the part that makes the meta-agent safe rather than cute — every agent in the repo ships with golden fixtures: a handful of known question/answer/diff cases with expected structured outputs, run in CI on any change to that agent. Then "agent that makes agents" has a real contract: the generated agent doesn't merge unless its own eval cases exist and pass, TDD enforced by the pipeline rather than by hoping the meta-agent remembered. The fixtures also catch upstream model drift, which you'll otherwise misdiagnose as everything else.

On plugins versus everything-in-repo: I'd drop the plugin question entirely, because it's not the boundary that matters. The real boundary is between the *deterministic shell* — audio capture, file I/O, git plumbing, index rebuilding — and the *generative interior* — agents, prompts, doc logic. The shell should be small, boring, human-written, and heavily tested; the interior is where self-modification and vibe-coded churn are cheap and safe. Whether the interior is "plugins" or "code in the repo" is a packaging detail; in-repo is cleaner, agreed.

And your instinct about the audio path deserves a stronger justification than "harder to test." Prioritize testing by *irreversibility*, not difficulty. Nearly everything in this system is regenerable from raw inputs — indexes rebuild, structured answers re-derive from transcripts, docs re-integrate. The raw capture path is the one component whose failure destroys something nothing can reconstruct: the recording that silently truncated, the transcript that dropped a segment nobody noticed. It's the single unrecoverable failure mode in the design, it fails silently, and it's exactly the kind of fiddly platform-dependent I/O code that LLM-generated code gets subtly wrong. That's three independent reasons it's the one place to be conservative: write it by hand, test it hard, and then never let the agents touch it.
